Settlement Auditability

An eight-item transparency checklist

For each disputed contract we report eight binary flags. The list is deliberately a transparent checklist, not a black-box score. Each flag has a one-sentence operational definition that maps to a single field in the underlying data.

The eight items

  1. Rule text available — Some recoverable natural-language statement of the resolution criteria exists, joined from pm_markets.question, episode_panel.question_text, or UMA ancillary_data_decoded.
  2. Named source specified — The recoverable text contains an external authority pattern: a URL, "according to / official(ly) / reported by / determined by / will resolve based on / sourced from", etc. Heuristic regex; not a hand-coded ground truth.
  3. Fallback specified — The text contains a fallback or contingency pattern: "if the source / fallback / in the event / if unavailable / otherwise".
  4. Edge cases specified — The text contains edge-case language: "edge case / N/A / cancelled / postponed / tied / abandoned / if not resolved".
  5. Settlement chain complete — A non-pending final payoff state is observed (final_payoff ≠ "Pending").
  6. Repeated-request flag — Chain type is classified as either Request-voiding chain or Repeated adapter-routed request (always true for this disputed sample by construction).
  7. Final payoff observed — Final payoff is non-null (Yes / No / Other / Invalid).
  8. Benchmark agreement — Final payoff agrees with the LLM-coded benchmark, OR the contract is not flagged as a candidate mismatch. Disagreement = the contract is in the 64-agreed candidate-mismatch set. This is a diagnostic flag, not an externally validated error count.

Items 1–4 are heuristic regex over a noisy text corpus. They tag whether the relevant language appears, not whether the contract was correctly designed. Treat the flags as scaffolding for manual inspection, not as a quality grade.
