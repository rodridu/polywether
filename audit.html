Settlement Auditability

Nine flags grouped into four categories

For each disputed contract, nine binary flags grouped into four categories that map to different kinds of failure. Transparent checklist, not a black-box score.

The categories

A. Text availability

Data we can see. Tells you whether anyone can audit the contract at all.

  1. Rule text recovered. Question text non-null in any of pm_markets.question / episode_panel.question_text / pm_resolution_events.ancillary_data_decoded.
  2. Description recovered. Polymarket's description field (longer resolution criteria) is non-null.
  3. UMA ancillary data recovered. The decoded ancillary data string from the on-chain UMA event is non-null.

B. Contract clarity (heuristic regex)

What the contract says about resolution. Items 4–6 fire if the relevant language appears in the recovered text — they don't certify correctness.

  1. Named external source language. Text contains "according to / official(ly) / determined by / verified by / sourced from / will resolve based on" or a URL.
  2. Fallback / contingency language. Text contains "if the source / fallback / in the event / if unavailable / otherwise".
  3. Edge-case language. Text contains "edge case / N/A / cancelled / postponed / tied / abandoned / if not resolved".

C. Settlement-chain observability

What the mechanism actually did. Deterministic from on-chain data.

  1. Final payoff observed. Final payoff state non-null and not Pending. ~99.5% of disputed sample.
  2. Multi-episode chain. Contract has at least two settlement requests against it.

D. Diagnostic disagreement

Where the LLM-coded benchmark disagrees with the final payoff. Diagnostic, not an externally validated error count.

  1. Broad candidate mismatch. Final payoff appears to disagree with a rule-implied label coded from contract text. 94 of 2,221 (4.2%) by the broad first-pass coding; 64 of those are agreed across both coding passes (category-level aggregate; per-row agreement not exposed here). Flag for inspection, not a verdict.

Removed from earlier version: separate "Settlement chain complete" flag (collapsed into Final payoff observed), and "Repeated-request flag" (always true for disputed sample by construction, non-informative).
