Settlement-risk tier

The screening tier on each disputed contract is a rules-based v0. Computed from existing audit flags + chain-type×category revision rates:

HighBroad candidate mismatch flagged, OR rule text not recoverable, OR (rule text present but no named source AND no fallback). 611 of 2,221 contracts (27.5%).
CautionNo fallback language, OR no edge-case language, OR same chain-type × category bucket has revision rate above the sample median. 1,513 of 2,221 (68%).
LowSource, fallback, edge-case language all present, and bucket revision rate at or below sample median. 92 of 2,221 (4%).
TestPolymarket dev/test market detected via JSON-blob ancillary data (e.g. {"earlyExpiration":1,"q":{"title":"Test propose only…"}}). 5 of 2,221. Excluded from the High watchlist + default Lookup view; filterable via the explicit Test chip.

The tier is a screening flag, not a prediction of correct/incorrect settlement. It says: the payoff depends on discretionary or ambiguous interpretation rather than a clean event-to-label mapping.

Look up a single contract by ID

Settlement Risk Monitor doesn't currently host a query API. Two ways to look up a contract programmatically:

  1. Direct URL. https://rodridu.github.io/polywether/contract.html?id={short_id} renders the per-contract page. Short IDs are 8-char hex (sha256 prefix of the raw condition_id).
  2. Fetch JSON, filter client-side. Pull disputed_contracts.json once (~5 MB) and filter by id, raw_condition_id (full 0x… 64-hex), slug, or substring of question_text.

Example: filter for High-tier contracts in JS:

const rows = (await (await fetch('/data/disputed_contracts.json')).json()).rows;
const highRisk = rows.filter(r => r.settlement_risk_tier === 'High');
console.log(highRisk.length, 'High-risk contracts');

Refresh cadence

Data refreshes via two paths:

  1. Aggregate refresh — automatic, weekly. Mondays 8am UTC + on push to data-source/**. The refresh-aggregates workflow regenerates settlement_funnel.json, chain_signatures.json, candidate_mismatches.json from committed source CSVs.
  2. Per-contract refresh — manual. Disputed-contract details, chain examples, and the duckdb-wasm DB are refreshed when the paper's analysis pipeline regenerates the rev3 panel. The pipeline lives in pipelines/.

Linked tools

SQL Lab (query.html), Audit (audit.html, nine-flag per-contract checklist), Chain View (chain.html), and Cases (cases.html) are also reachable from the footer. Source repo: github.com/rodridu/polywether.

Sample frame & the negRisk gap

The disputed-contract panel (n=2,221) is the condition_id-linked subset of disputed Polymarket-via-UMA settlement events. Markets without a recoverable condition_id in the rev3 panel are out of scope — including some that resolve via Polymarket's negRisk mechanism (e.g. Paris weather contracts and some prediction-market families that use a negRisk multi-outcome adapter rather than the standard binary condition_id path).

If you query the SQL Lab and don't find a Polymarket market you remember (e.g. a Paris-temperature-on-day-X contract), the negRisk linkage is the most likely reason. The site does not currently bridge negRisk-resolved markets back to pm_markets.condition_id. The paper's §3.1 caveat covers this gap; expanding the panel to include negRisk-side identifiers is a separate ingestion track (not in scope for the v0 dashboard).

Practical effect: the 2,221 disputed-contract figure is a strict undercount of all disputed Polymarket settlements. It is an exact count for the auditable-via-condition_id subpopulation; the negRisk-resolved subset is unmeasured here.

Benchmark validation transparency

The candidate-mismatch layer is built from a two-pass LLM-coded benchmark over recoverable contract text. Pass 1 flags 94 broad candidate mismatches (rows where classification == 'uncorrected_oracle_error'). Pass 2 reconciles a subset.

Row-level fields now exposed

candidate_mismatches.json.rows[] exposes for every one of the 94:

  • id, raw_condition_id, question_text, category
  • first_proposal (P_i1), final_payoff (S_i), coded_benchmark (G_i_label)
  • operative_proposed, operative_final
  • stratum — two_benchmark if both passes independently coded this specific row (n=55); frozen if assignment carried forward from a single pass (n=39).
  • G_i_source
  • mechanical_final_correct_a1, legacy_final_correct — replication-grade per-row correctness flags from the paper's reconciliation pipeline.
  • mismatch_agreed — boolean equivalent to (stratum == 'two_benchmark'). Per-row, 55 of 94 are agreed.

What is not yet exposed

  • Per-row LLM rationales (free-text reasoning behind each flag).
  • Exact prompts used in pass 1 vs pass 2.
  • Blinded third-pass independent validation against an external human coder.
  • Sensitivity of the 94 count to rule-text recovery — i.e. how many flagged rows depend on edited or partial rule text.

Rationales and prompts live in the paper appendix repository; they are not in the public dashboard pending an independent-coder validation pass. Treat the 94 number as a diagnostic measurement over the disputed-contract panel — not a population-wide error rate, and not a final adjudication of any specific contract.

Paper's "64 agreed" figure

Paper Table 1 reports 64 mismatches agreed at the category level. This is a coarser aggregate than per-row stratum agreement: it asks whether per-category counts agree across passes, not whether a specific row was independently coded twice. Per-row stratum agreement (55) is what referees should use for replication-grade auditing of individual contracts. Both numbers come from the same two-pass benchmark; they answer different questions.

The paper's design does not identify full-population accuracy, trader losses, causal effects, or cross-platform generalizability.
