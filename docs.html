Settlement-risk tier

The screening tier on each disputed contract is a rules-based v0. Computed from existing audit flags + chain-type×category revision rates:

HighBroad candidate mismatch flagged, OR rule text not recoverable, OR (rule text present but no named source AND no fallback). 616 of 2,221 contracts (28%).
CautionNo fallback language, OR no edge-case language, OR same chain-type × category bucket has revision rate above the sample median. 1,513 of 2,221 (68%).
LowSource, fallback, edge-case language all present, and bucket revision rate at or below sample median. 92 of 2,221 (4%).

The tier is a screening flag, not a prediction of correct/incorrect settlement. It says: the payoff depends on discretionary or ambiguous interpretation rather than a clean event-to-label mapping.

Static JSON endpoints

All data files live under /data/ and are committed to the repo. Treat them as a static API: paths and field names are stable; refresh cadence is documented below.

/data/disputed_contracts.jsonOne row per disputed contract (n=2,221). Reader-facing fields: id, raw_condition_id, slug, question_text, description, ancillary_text, category, first_proposal, final_payoff, revised, multi_episode, chain_type, candidate_mismatch, settlement_risk_tier, risk_reasons, first_proposal_time, final_settlement_time, audit{ rule_text_present, description_present, ancillary_present, named_source_present, fallback_present, edge_cases_present, final_payoff_observed, multi_episode, candidate_mismatch }
/data/settlement_funnel.json9-step funnel ladder + summary numbers (paper Table 1).
/data/chain_signatures.json23 raw chain-pattern aggregates + 3-row chain_type_summary.
/data/candidate_mismatches.json94 broad mismatches; 64 agreed at category level. Per-row agreement not exposed.
/data/conditional_rates.jsonPre-computed revision + candidate-mismatch rates by 5 dimensions (chain_type, category, first_proposal, named_source presence, fallback presence) + overall.
/data/public_cases.json7 named cases with status, market matches, and disputed-sample matches. Locate, do not adjudicate.
/data/chain_examples.json10 hand-curated example contracts for Chain View, sports-only, with full request-by-request timelines.
/data/headline_statistics.jsonPaper-level headline statistics (richer schema than the funnel summary).
/data/settlement_explorer.duckdbStripped DuckDB (10 MB) for in-browser queries via duckdb-wasm. Tables: contracts (2,221 rows), episodes (2,668), funnel, chain_signatures, flip_decomposition.

Look up a single contract by ID

Settlement Risk Monitor doesn't currently host a query API. Two ways to look up a contract programmatically:

  1. Direct URL. https://rodridu.github.io/polywether/contract.html?id={short_id} renders the per-contract page. Short IDs are 8-char hex (sha256 prefix of the raw condition_id).
  2. Fetch JSON, filter client-side. Pull disputed_contracts.json once (~5 MB) and filter by id, raw_condition_id (full 0x… 64-hex), slug, or substring of question_text.

Example: filter for High-tier contracts in JS:

const rows = (await (await fetch('/data/disputed_contracts.json')).json()).rows;
const highRisk = rows.filter(r => r.settlement_risk_tier === 'High');
console.log(highRisk.length, 'High-risk contracts');

Refresh cadence

Data refreshes via two paths:

  1. Aggregate refresh — automatic, weekly. Mondays 8am UTC + on push to data-source/**. The refresh-aggregates workflow regenerates settlement_funnel.json, chain_signatures.json, candidate_mismatches.json from committed source CSVs.
  2. Per-contract refresh — manual. Disputed-contract details, chain examples, and the duckdb-wasm DB are refreshed when the paper's analysis pipeline regenerates the rev3 panel. The pipeline lives in pipelines/.

Linked tools

SQL Lab (query.html), Audit (audit.html, nine-flag per-contract checklist), Chain View (chain.html), and Cases (cases.html) are also reachable from the footer. Source repo: github.com/rodridu/polywether.

Versioning

JSON schemas are stable in additive sense: new fields may be added; existing field names will not change without a deprecation note in the repo CHANGELOG. The settlement_risk_tier rule is v0; future versions may incorporate continuous risk scores rather than three buckets.
